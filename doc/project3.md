## 1. 서론

### 1.1 문서 목적 및 범위
본 프로젝트 계획서는 안전한 사내 협업을 위한 경량 클라우드 파일 공유 시스템인 '미니 드라이브(Mini Drive)'에 대한 전반적인 내용을 정리하여 프로젝트를 진행하는 데 있어 각 세부 사항의 지표로 삼는 것을 목적으로 한다.

본 문서는 다음과 같은 범위의 지표로 적용할 수 있다.
① 프로젝트 개요 (프로젝트 정의, 주요 기능 정리)
② 개발 계획 및 환경 (개발 절차 모형, 내용 및 일정, 소프트웨어/하드웨어 환경)
③ 팀 구성 (팀 구조/역할 및 책임)
④ 품질관리 (팀 미팅, 변경 사항 및 산출물 관리)
⑤ 산출물 (산출물 정의, 작성일 및 담당자)

### 1.2 프로젝트 개요

#### 1.2.1 프로젝트 정의
조직 내 여러 팀이 문서를 중앙 집중식으로 보관하고 안전하게 공유할 수 있도록 지원하는 웹 기반의 클라우드 파일 관리 시스템을 구축한다. 복잡한 기능은 덜어내고 직관적인 UI를 제공하여 별도의 학습 없이 협업할 수 있는 환경을 제작한다.

#### 1.2.2 주요 기능 설명

| ID | 설명 |
| :--- | :--- |
| **F01** | 다양한 포맷의 파일 업로드/다운로드, 계층형 폴더 관리 및 휴지통 기능 제공 |
| **F02** | 조직 내 사용자 간 접근 권한 세분화 및 외부 공유용 만료 기간 설정 링크 생성 |
| **F03** | 파일명, 날짜, 사용자, 파일 유형 등을 조합한 다중 필터 적용 파일 검색 기능 |
| **F04** | 동일 파일 수정 시 이전 버전 보관 및 특정 과거 버전으로의 복원 기능 지원 |
| **F05** | 이메일 기반 로그인, 데이터 전송 구간 암호화 및 사용자별 저장 공간(Quota) 할당 |

### 1.3 프로젝트 목표
복잡한 엔터프라이즈 기능 대신 필수 핵심 기능에 집중하여, 누구나 3분 이내에 숙달할 수 있는 직관적인 파일 공유 환경을 제공한다. 또한, "Lean & Fast Architecting" 전략을 통해 낙관적 락(Optimized Locking) 및 메타데이터 중심의 인메모리 캐싱을 도입함으로써 서버 리소스를 최소화하고, 소규모 팀이 별도의 인프라 엔지니어 없이도 저비용 고효율로 안정적인 운영을 할 수 있는 사내 협업 시스템을 완성하는 것을 목표로 한다.

### 1.4 용어 정의
본 문서의 이해를 돕기 위해 사용된 모든 용어 및 약어를 설명하고 정의합니다.

| 용어 | 설명 |
| :--- | :--- |
| **낙관적 락 (Optimized Locking)** | 데이터 갱신 시 충돌이 발생하지 않을 것이라고 가정하고, 마지막에 충돌 여부를 확인하여 처리하는 동시성 제어 방식 |
| **Quota** | 시스템 내에서 개별 사용자 또는 그룹에게 할당된 데이터 최대 저장 공간 한도 |
| **TCO (Total Cost of Ownership)** | 소프트웨어의 개발 및 도입부터 운영, 유지보수, 폐기까지 발생하는 총 소유 비용 |
| **Redis** | 파일 검색 속도 향상을 위한 메타데이터 캐싱 목적으로 활용되는 고성능 인메모리 데이터 구조 저장소 |

## 2. 개발 계획

### 2.1 개발 절차 모형

**1) 소프트웨어를 개발하기 위한 개발 절차 모형**
초기 핵심 요구사항(5대 기능)은 정의되어 있으나, UI/UX의 직관성과 사용성 측면에서 잦은 요구사항 변경과 피드백이 발생할 것으로 예상된다. 팀원들 간의 의사소통이 원활하게 이루어지고, 사용자의 요구사항을 빠르게 반영하여 "Lean & Fast" 전략을 달성할 수 있도록 Agile 방법론 중 SCRUM 모델을 사용하기로 결정하였다.

**2) 개발 절차 모형과 함께 분석, 설계를 진행하기 위한 방법론**
여러 가지 모델링 개념과 캡슐화를 잘 지원하는 객체지향 방법론을 채택함과 동시에, 서버 리소스를 효율적으로 관리하기 위해 모듈형 모놀리스(Modular Monolith) 아키텍처 설계를 병행한다. 이는 클래스와 모듈이 자연스럽게 소프트웨어의 한 단위가 되어 분석, 테스트, 유지보수, 확장(Scale-out) 등에 용이한 장점을 제공한다.

## 3. 팀 구성

### 3.1 팀 구조
팀장의 역할은 존재하나, 모든 멤버가 상호 협력하고 서로의 개발 활동을 확인하며 진행하는 수평적 팀 구조를 채택하였다. 특히 'Lean & Fast Architecting' 전략을 달성하기 위해 기능별 모듈 개발 시 팀원 간의 긴밀한 피드백이 가능한 구조로 운영한다.

### 3.2 역할 및 책임

| 이름 | 역할 | 책임 (공동 / 개별) |

## 4. 품질 관리

### 4.1 팀 미팅 계획

**(1) Sprint Planning Meeting**
- **팀 미팅 시간** : 매주 금요일 6시
- **토의 내용** : 다음 주차 Sprint 목표 설정 및 Sprint Backlog(파일 관리, 공유 권한 등) 계획 수립
- **미팅 참석자** : 팀원 전체, 팀장

**(2) Daily Scrum Meeting**
- **팀 미팅 시간** : 매일 오후 6시에 30분간 진행
- **토의 내용** : 어제 한 일, 오늘 할 일, 개발 중 발생한 장애 현상(기술적 부채 등) 공유
- **미팅 참석자** : 팀원 전체, 팀장

**(3) Sprint Review**
- **팀 미팅 시간** : Sprint 종료 시점 (2주 단위)
- **토의 내용** : Sprint 기간 동안 개발된 기능(예: 파일 업로드, 검색 필터 등) 리뷰 및 시연
- **미팅 참석자** : 팀원 전체, 팀장

**(4) Sprint Retrospective**
- **팀 미팅 시간** : Sprint 종료 시점
- **토의 내용** : 팀 내부 프로세스 회고, 잘된 점, 개선할 점(코드 품질, 협업 방식 등) 도출

**(5) 비정기 회의**
- 프로젝트 진행에 심각한 리스크(보안 결함, 일정 지연 등)가 발생한 경우 진행된다.
- 팀장 또는 지도교수에 의해 요청될 수 있으며, 문제 해결을 위한 집중 토의를 수행한다.
- **미팅 참석자** : 팀원, 팀장, (지도교수, 멘토)

### 4.2 변경사항 관리 방법
- 기능 변경(예: 기술 스택 변경, 기능 추가 등)을 위해 팀원 전체가 권한을 가지며 다수결의 합의가 필요하다.
- 변경사항 발생 시 발생 전-후의 설계 및 코드 변경 내용을 반드시 Git 이력 및 문서 기록물로 남긴다.
- 관련 산출물(요구사항 정의서 등)은 수정 즉시 팀원 전체가 공유 클라우드 및 GitHub를 통해 소지한다.
- 의견 충돌로 진행이 멈출 시에는 프로젝트 완수와 일정 준수를 위해 팀장이 좀 더 강력한 발언권을 갖는다.
- 중요 변경 사항(아키텍처 변경 등) 결정 시 지도 교수님께 보고하여 피드백을 받으며, 경우에 따라 재변경될 수 있다.

### 4.3 산출물 관리 방법

**(1) 산출물에 대한 명명(Naming) 방법**
- **소프트웨어** : 버전_해당 버전의 산출 일자 (ex. MiniDrive_v1.0_20241101.zip)
- **문서** : [팀명]문서명_YYMMDD_Doc-해당 문서의 버전 (ex. [김태림]프로젝트관리계획서_241025_Doc-001.pdf)
- **그 외 파일** : 해당 파일의 목적이 명확하게 드러나는 이름 (ex. 241025_UI_Design_Draft.fig)

**(2) 산출물의 버전 관리 방법 및 버전관리 담당자**
- **소프트웨어** : GitHub 리포지토리를 사용하여 브랜치별로 관리하며, 배포 시점에 Tag를 생성한다. (담당자: 팀원 전체)
- **문서** : 파일명 뒤에 날짜를 기재하고, 문서 내 '제/개정 이력' 테이블을 반드시 수정한다. (담당자: 팀장)
- **그 외 파일** : 파일명 앞에 날짜를 붙이며 중요도에 따라 순번을 추가한다. (담당자: 전체)

**(3) 산출물의 저장 방법**
- **소프트웨어** : GitHub (Public 또는 Private Repository)
- **문서** : 구글 드라이브(팀 공유 폴더) 및 GitHub 내 Documentation 폴더
- **그 외 파일** : 팀원 공유 클라우드 서버 (Google Drive 등)
