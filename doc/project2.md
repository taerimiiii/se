# Project II: 대상 시스템 품질 요소 추정하기

요구사항 명세서(CRD)와 1장의 시스템 정의서를 바탕으로, '미니 드라이브(Mini Drive)' 시스템의 주요 이해관계자(Stakeholder)들이 요구하는 핵심 품질 요소를 다음과 같이 추정하고 정의합니다.

## (1) 사용자가 원하는 기대치는 무엇인가? (외부 품질 중심)
실제 시스템을 사용하는 일반 직원들은 업무 효율성과 직결되는 **외부 품질(External Quality)**에 가장 큰 관심이 있습니다.

* **신뢰성 (Reliability):** 파일 업로드나 다운로드 중 시스템이 멈추거나 고장 나지 않고 항상 정상적으로 동작해야 합니다.
* **사용성 (Usability):** 새로운 시스템 도입에 따른 학습 곡선을 최소화하기 위해, 별도의 매뉴얼 없이도 직관적이고 사용하기 쉬운 UI/UX가 제공되어야 합니다.
* **성능 효율성 (Performance Efficiency):** 수십 MB 이상의 대용량 설계 파일이나 영상 파일도 지연 없이 빠르게 다운로드되고, 다중 필터 검색 결과가 즉각적으로 표출되어야 합니다.

## (2) 개발자 혹은 유지보수 담당자가 원하는 기대치는 무엇인가? (내적 품질 중심)
시스템을 직접 구축하고 향후 지속해서 관리해야 하는 개발/운영팀은 소프트웨어의 수명과 안정성을 결정짓는 **내부 품질(Internal Quality)**에 관심이 있습니다.

* **가독성 (Readability):** 여러 개발자가 협업하거나 담당자가 변경되더라도, 코드를 이해하기 쉽도록 명확한 코딩 표준과 구조를 갖추어야 합니다.
* **유지보수성 (Maintainability):** 향후 요구사항 명세서 11항에 명시된 '문서 공동 편집'이나 'AI 기반 검색'과 같은 새로운 확장 기능을 기존 시스템과 충돌 없이 추가하기 쉬운 모듈화된 구조여야 합니다.
* **검증 가능성 (Testability/Verifiability):** 동시 다발적인 파일 업로드나 복잡한 권한 설정 시 발생할 수 있는 오류를 자동화된 방식으로 쉽게 테스트하고 검증할 수 있어야 합니다.

## (3) 개발 조직의 관리자(스폰서)가 원하는 기대치는 무엇인가? (비즈니스 및 보안 중심)
프로젝트에 예산을 투입하고 총괄 책임을 지는 스폰서/관리자는 조직의 **비즈니스 가치 보호와 비용 절감**에 가장 큰 관심이 있습니다.

* **비용 효율성 (Cost Effectiveness):** 불필요하게 복잡한 기능(예: 자체 실시간 동시 편집 엔진 개발 등)을 배제하고, 핵심 요구사항에 집중하여 한정된 예산과 기간 내에 적은 비용으로 성공적으로 개발되기를 원합니다.
* **보안성 (Security):** 조직의 중요한 데이터가 중앙 집중식으로 보관되므로, 외부 해킹이나 정보 유출 방지가 최우선입니다. 특히 외부 협력업체와 만료 기간이 있는 링크로 파일을 공유할 때, 허가되지 않은 접근을 완벽히 차단하여 데이터 기밀성과 무결성을 보장해야 합니다.

---

> 💡 **종합 요약:** > 미니 드라이브 프로젝트는 관리자가 요구하는 **보안성**의 토대 위에서, 사용자를 만족시키는 **사용성 및 성능 효율성**을 제공해야 합니다. 이를 지속 가능하게 만들기 위해 개발 단계부터 높은 수준의 **유지보수성**을 확보하는 것이 본 프로젝트의 핵심 품질 목표입니다.
