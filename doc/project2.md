# Project II: 대상 시스템 품질 요소 추정하기

요구사항 명세서(CRD)와 시스템 정의서를 바탕으로, '미니 드라이브(Mini Drive)' 시스템의 주요 이해관계자들이 요구하는 핵심 품질 요소를 정량적 기준을 포함하여 다음과 같이 정의합니다.

## (1) 사용자가 원하는 기대치는 무엇인가? (외부 품질 중심)
실제 시스템을 사용하는 일반 직원들은 업무 효율성과 직결되는 외부 품질(External Quality)에 가장 큰 관심이 있습니다.

* **신뢰성 (Reliability):** 시스템은 연간 **99.9% 이상의 가동률(Availability)**을 보장해야 합니다. 파일 업로드/다운로드 중 네트워크 순단이 발생하더라도 **자동 재시도(Retry) 메커니즘**을 통해 데이터 유실 없이 작업을 완료해야 합니다.
* **사용성 (Usability):** 신규 사용자가 별도의 매뉴얼 없이도 파일 업로드 및 공유 기능을 **3분 이내**에 완수할 수 있도록 직관적인 UI를 제공합니다. 모든 인터페이스는 3-Click 이내에 원하는 기능에 도달할 수 있는 구조를 지향합니다.
* **성능 효율성 (Performance Efficiency):** 100MB 이하 파일 기준, 업로드/다운로드 시 **응답 지연 시간(Latency)을 3초 이내**로 유지합니다. 다중 필터 검색 시 결과 리스트는 **1.5초 이내**에 화면에 출력되어야 합니다.

## (2) 개발자 혹은 유지보수 담당자가 원하는 기대치는 무엇인가? (내적 품질 중심)
시스템을 구축하고 운영하는 팀은 소프트웨어의 수명과 안정성을 결정짓는 내부 품질(Internal Quality)에 관심이 있습니다.

* **가독성 (Readability):** 표준 코딩 규격(예: Google Java Style Guide)을 준수하며, 정적 분석 도구 활용 시 **중복 코드 비율을 5% 미만**으로 유지하여 코드 이해도를 높입니다.
* **유지보수성 (Maintainability):** 기능 간 결합도를 낮춘 **마이크로서비스 또는 계층형 아키텍처**를 채택합니다. 이를 통해 'AI 기반 검색' 등 향후 확장 기능 추가 시 기존 기능에 미치는 영향을 최소화하고 모듈 단위의 독립적 수정이 가능해야 합니다.
* **검증 가능성 (Testability):** 핵심 비즈니스 로직에 대해 **유닛 테스트 커버리지를 80% 이상** 확보합니다. 모든 코드는 CI/CD 파이프라인 내에서 자동화된 테스트를 통과해야만 배포 가능하도록 설정합니다.

## (3) 개발 조직의 관리자(스폰서)가 원하는 기대치는 무엇인가? (비즈니스 및 보안 중심)
프로젝트의 예산과 비즈니스 가치를 총괄하는 관리자는 비용 절감과 데이터 보안에 가장 큰 관심이 있습니다.

* **비용 효율성 (Cost Effectiveness):** 클라우드 자원 사용량을 모니터링하여 설정된 **월간 운영 예산 범위(예: 상한선 대비 90%)** 내에서 시스템이 구동되도록 최적화합니다. 불필요한 자체 엔진 개발보다는 검증된 오픈소스를 활용하여 개발 기간을 단축합니다.
* **보안성 (Security):** 모든 데이터 전송은 **TLS 1.3 암호화**를 적용하며, 외부 공유 링크 생성 시 **비밀번호 설정 및 최대 7일의 유효 기간**을 강제 설정합니다. 관리자 페이지는 인가된 IP에서만 접근 가능하도록 제한하여 데이터 무결성을 보장합니다.

---

> 💡 **종합 요약:** > 미니 드라이브 프로젝트는 관리자가 요구하는 **보안성(암호화 및 접근 제어)**의 토대 위에서, 사용자를 만족시키는 **정량적 성능 지표(응답 시간 3초 이내)**를 달성해야 합니다. 이를 위해 개발 단계부터 **테스트 커버리지 80%** 및 **모듈화된 구조**를 확보하여 유지보수성을 극대화하는 것이 본 프로젝트의 핵심 목표입니다.
